The Output Panel displays results from script execution, with different views depending on the selected mode.

## Panel Layout

The Output Panel sits below the code editor and includes:

- **Tab selector** for different output views
- **Execution status** badge showing run state
- **Duration** timer for execution time
- **Controls** for copy, fullscreen, and clear

Drag the divider between the editor and output panel to resize.

## Output Tabs

<Tabs>
  <TabItem label="Raw Output" icon="document">
    The complete text output from your script execution:

    - Full stdout from the collector
    - Includes warnings, errors, and debug messages
    - Preserves formatting and line breaks
    - Displayed in monospace font

    <img src="/lmda-composer/core-features/output-raw.png" alt="Raw output tab" style="border-radius: 0.5rem; margin: 1rem 0;" />

    <Aside type="tip" title="Copy Output">
      Click the copy button or use the keyboard shortcut to copy the entire output to your clipboard.
    </Aside>
  </TabItem>

  <TabItem label="Parsed" icon="list-format">
    Structured view of output based on the current execution mode.

    **Active Discovery Mode:**

    | Column | Description |
    |--------|-------------|
    | Instance ID | The unique identifier (wildvalue) |
    | Instance Name | Display name for the instance |
    | Description | Optional description text |
    | Properties | Auto-properties as key=value pairs |

    The header shows total count: "Parsed (15)" for 15 instances.

    **Collection Mode:**

    | Column | Description |
    |--------|-------------|
    | Datapoint | The datapoint name |
    | Value | The numeric value |

    **Batch Collection Mode:**

    | Column | Description |
    |--------|-------------|
    | Instance | The wildvalue/instance ID |
    | Datapoint | The datapoint name |
    | Value | The numeric value |

    <img src="/lmda-composer/core-features/output-parsed-ad.png" alt="Parsed output showing Active Discovery instances" style="border-radius: 0.5rem; margin: 1rem 0;" />
  </TabItem>

  <TabItem label="Validation" icon="approve-check-circle">
    Real-time validation feedback based on the current mode's output rules.

    **Status Indicators:**

    <div style="display: flex; gap: 1rem; flex-wrap: wrap; margin: 1rem 0;">
      <StatusBadge status="success" label="All Valid" />
      <StatusBadge status="warning" label="Warnings" />
      <StatusBadge status="error" label="Errors" />
    </div>

    The tab header shows a summary: checkmark when valid, or warning/error counts.

    <img src="/lmda-composer/core-features/output-validation.png" alt="Validation tab showing output validation results" style="border-radius: 0.5rem; margin: 1rem 0;" />
  </TabItem>

  <TabItem label="Graph" icon="random">
    For **TopologySource** scripts, displays an interactive network topology graph with full visualization capabilities.

    **Node Types:**
    | Type | Color | Description |
    |------|-------|-------------|
    | Device | Blue | Vertices defined in your topology output |
    | External | Gray | Resources referenced in edges but not in vertices |

    **Edge Type Colors:**
    | Type | Color |
    |------|-------|
    | Network | Cyan |
    | Bridge | Purple |
    | CDP | Orange |
    | LLDP | Green |

    **Interactive Features:**
    - **Pan & Zoom** — Use mouse wheel to zoom, drag to pan the canvas
    - **Node Selection** — Click any node to view its details in the side panel
    - **Edge Selection** — Click any edge to see connection info, health data, and metadata
    - **Details Panel** — Collapsible side panel shows properties for selected elements
    - **Dynamic Legend** — Shows only edge types and node types present in your topology

    **Node Details Panel:**
    - Node label and type
    - Custom properties from your script

    **Edge Details Panel:**
    - Source and target connection
    - Edge type and display type
    - Instance information (from/to instances)
    - Health DataSources with Tx/Rx datapoints
    - Custom metadata
    - Metric reporting node

    <Aside type="note">
      The Graph tab only appears when running in TopologySource-compatible mode. The topology data is parsed from your script's vertex and edge output.
    </Aside>
  </TabItem>
</Tabs>

## Validation Rules by Mode

<Tabs>
  <TabItem label="Active Discovery" icon="seti:todo">
    | Issue | Severity | Description |
    |-------|----------|-------------|
    | Empty instance ID | <Badge text="Error" variant="danger" /> | Instance ID is required |
    | ID exceeds 1024 chars | <Badge text="Error" variant="danger" /> | Maximum length exceeded |
    | Invalid ID characters | <Badge text="Error" variant="danger" /> | Contains spaces, `=`, `:`, `\`, or `#` |
    | Name exceeds 255 chars | <Badge text="Warning" variant="caution" /> | May be truncated |
    | Duplicate instance ID | <Badge text="Warning" variant="caution" /> | Same ID appears multiple times |
  </TabItem>

  <TabItem label="Collection / Batch" icon="heart-pulse">
    | Issue | Severity | Description |
    |-------|----------|-------------|
    | Invalid format | <Badge text="Error" variant="danger" /> | Line doesn't match `key=value` |
    | Non-numeric value | <Badge text="Error" variant="danger" /> | Value must be a number |
    | Invalid datapoint name | <Badge text="Warning" variant="caution" /> | Contains special characters |
    | Missing wildvalue (Batch) | <Badge text="Error" variant="danger" /> | Batch lines need `instance.datapoint=value` |
    | Non-matching lines | <Badge text="Warning" variant="caution" /> | Lines that don't match expected format |
  </TabItem>

## Error Display

When a script fails, the Raw Output tab shows:

<Aside type="caution" title="Common Error Types">
  - **Compilation errors** – Groovy/PowerShell syntax issues
  - **Runtime exceptions** – Errors during execution (null pointer, type mismatch)
  - **Timeout errors** – Script exceeded the collector's time limit
  - **Connection failures** – Network issues reaching the target device
</Aside>

Error messages appear in red text with stack traces and diagnostic information. Partial output may appear if the script produced output before failing.

## Best Practices

### Debugging with Output

1. Use `println` statements liberally during development
2. Print variable values and intermediate results
3. Once logic is verified, format output for the target mode

### Reading Validation Results

1. Check the Validation tab after each run
2. Fix all **errors** before deploying to production
3. Address **warnings** when possible—they indicate potential issues

### Working with Large Output

- Use **fullscreen mode** for better readability
- The **copy button** captures all content, even if display is truncated
- Add summary lines at the end of your script for quick verification
